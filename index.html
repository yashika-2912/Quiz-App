HTML · DOM STRUCTURE

The Human Compiler

Students look at the code — sketch what they think the browser renders. Then reveal surprises one by one.

1:00
// Code shown on smartboard
<!-- Students: what does this render? -->
<section>
  <div>
    <h1>Hello</h1>
    <img />
    <p></p>
    <button disabled>Click Me</button>
  </div>
</section>
<footer>
  <span>
    Done
    <div>I am inside a span</div>
  </span>
</footer>
Q1
How many visible things actually appear on screen?
3 things: "Hello" (h1) + "Click Me" button (greyed out) + "Done" text.
→ Empty <p> renders nothing. <img> with no src shows a broken icon in some browsers, nothing in others.
Q2
Can the user click the button? Does it still exist in the DOM? ⚡ TWIST
No click allowed — disabled attribute blocks interaction.
Yes — it absolutely exists in the DOM. You can still select it with JavaScript.
→ disabled ≠ removed. It's just non-interactive.
Q3
Is this HTML valid? Find the illegal nesting. ⚡ TWIST
Invalid! A <div> is inside a <span>.
→ <span> is inline — it cannot contain block-level elements like <div>.
Q4
What does the browser actually do with the <div> inside the <span>? ⚡ TWIST
Auto-corrects it silently. The browser pushes the <div> outside the <span> and closes the <span> early.
→ The rendered DOM looks completely different from the written HTML. This is why valid HTML matters — browsers guess, and they don't always guess right.

💉1
Type <h1>HACKED</h1> — what happens and why?
A giant H1 renders on the page.→ innerHTML treats the input as actual HTML and parses it. It doesn't escape anything.
💉2
The img injection runs JavaScript — what triggered it?
onerror event handler. src="x" fails, browser fires onerror, which runs arbitrary JS.→ You don't need <script> to run JS. Event attributes are execution vectors too.

What's the real-world danger of injecting a password input into a real page? ⚡ TWIST
Phishing. The user thinks they're on a trusted site and types their real password into a fake field controlled by an attacker.→ This is exactly how stored XSS attacks work on real websites.

Change innerHTML → textContent→ textContent treats everything as plain text. HTML tags are displayed as literal characters, not parsed.

// Question 1 — Score Tracker (Type Coercion)
<input id="num" placeholder="Enter a number">
<button onclick="play()">Play</button>
<p id="out"></p>
<script>
  let score = 1;
  function play() {
    let val = document.getElementById("num").value;
    score += val;
    document.getElementById("out").textContent = score;
  }
</script>
What will be the output if input is 10?
›110
String concatenation! .value always returns a string. score starts at 1 (number), so 1 + "10" = "110" — not 11!→ Fix: wrap with Number(val) or parseInt(val). The classic type coercion trap.
// Question 2 — Object Reference
let a = { value: 10 };
let b = a;
a = { value: 20 };
console.log(b.value);
OUTPUT:
›10
b still points to the original object. b = a copies the reference, not the value. Reassigning a to a new object doesn't affect b at all.→ Reference vs value — b is still locked to { value: 10 }.
// Question 3 — Recursion Trace
<button onclick="run()">Run</button>
<script>
  function run() {
    function f(n) {
      if (n > 3) return n;
      return f(n + 1);
    }
    console.log(f(1));
  }
</script>
OUTPUT:
›4
f(1) → f(2) → f(3) → f(4). When n=4, n > 3 is true, returns 4. That value bubbles all the way back up.→ Trace it step by step: each call waits for the next until the base case hits.

Guess the Output

Students submit answers via QR before you reveal. Click each output line to unblur the answer.

1:00
// Question 1 — ASI Trap
function foo() {
  return
  {
    value: 42
  }
}
console.log(foo());
OUTPUT:
› undefined
JS inserts a semicolon after return because of the newline — Automatic Semicolon Insertion. The object literal below it is never reached.→ Fix: put the opening brace on the same line as return.
// Question 2 — Closure Loop
const funcs = [];
for (let i = 0; i < 3; i++) {
  funcs.push(() => i * i);
}
console.log(
  funcs[0](), funcs[1](), funcs[2]()
);

// Now change let → var. Output?
OUTPUT (let vs var):
let →0 1 4
var →9 9 9
let creates a new binding per iteration — each closure captures its own i. var is one shared i across all iterations. By the time functions run, var i is already 3, so 3×3=9.→ This is the classic closure-in-loop trap. let fixes it.
// Question 3 — Array Holes + Coercion
console.log([1,2,3] + [4,5,6]);

const arr = [1, , , 4];

console.log(arr.length);
console.log(arr[1]);
console.log(1 in arr);
console.log(3 in arr);
OUTPUT:
›"1,2,34,5,6"
›4
›undefined
›false
›true
[1,2,3]+[4,5,6] → arrays coerce to strings first: "1,2,3"+"4,5,6" = string concat, not math! Then for arr holes: arr[1] is undefined but 1 in arr is false — the slot doesn't exist at all. Length tracks highest index+1, index 3 (value 4) does exist so 3 in arr is true.→ Two of JS's darkest corners in one card: type coercion and array holes.
